# Tech Writing One — Reference

Source: https://developers.google.com/tech-writing/one

## Key principles

### Words and terminology
- Define new terms on first use or link to an existing definition
- Use the same term consistently throughout the document — don't synonymize
- Acronyms: on first use, write the full form + acronym in parentheses, bolding both: **Protocol Buffers** (**PB**)
- Only define an acronym if: (a) it is significantly shorter than the full form AND (b) it appears multiple times
- Pronouns (it, they, this, that) — place within 5 words of the noun they refer to; when in doubt, repeat the noun instead of using a pronoun

### Active vs. passive voice
- Active pattern: **subject + verb + object** ("The compiler generates an error")
- Passive pattern: object + form of "be" + past participle ("An error is generated by the compiler")
- Active voice: fewer words, clear who performs the action, easier to read
- Passive is acceptable when the actor is unknown or irrelevant
- Imperative sentences ("Open the file", "Click Save") are active — the implied subject is "you"

### Strong verbs
- Avoid generic verbs: is/are/was, occur, happen
- Replace with a strong, precise verb:
  - "An error occurs" → "The system raises an error"
  - "There is a bug" → "The component contains a bug"
  - "We are careful" → "We carefully validate"
- Avoid "There is / There are" — move the real subject to the front of the sentence

### Short, clear sentences
- One sentence = one idea (like one instruction in code)
- Replace wordy phrases with concise ones:
  - "at this point in time" → "now"
  - "is able to" → "can"
  - "provides a detailed description of" → "describes"
  - "causes the triggering of" → "triggers"
- Subordinate clauses: keep them only when they extend the main idea; if they shift topic → new sentence
- **That** (no comma) = essential clause: "calculations that avoid linear algebra"
- **Which** (with comma) = non-essential clause: "Python, which Guido invented, ..."
- Avoid subjective adjectives and adverbs; replace with numerical data:
  - "runs extremely fast" → "runs 225% faster"

### Lists and tables
- **Bulleted list** = unordered items; rearranging items does not change meaning
- **Numbered list** = steps or sequences; rearranging items changes meaning
- **Parallelism** — all list items must share the same grammatical structure (e.g., all start with a verb)
- Numbered lists: start each item with an imperative verb ("Open", "Configure", "Run")
- Every list and table requires an introductory sentence ending with a colon:
  - "The component accepts the following props:"
- Tables: short cells (max. 2 sentences), consistent data types per column, column headers required

### Paragraphs
- The first sentence of a paragraph = most important; it must state the paragraph's topic
- One paragraph = one topic; remove sentences that stray from the topic
- Length: 3–5 sentences (optimal); avoid "walls of text" (>7 sentences)
- Paragraph framework: **What** (information) → **Why** (significance) → **How** (application)

### Audience
- Formula: `documentation = (knowledge the audience needs) − (knowledge they already have)`
- Identify roles, experience level, familiarity with the project
- Avoid the "curse of knowledge" — don't assume the reader knows what you know
- Use plain language (many readers use translation software); avoid idioms and cultural metaphors
- Avoid sports or cultural references (baseball, cricket) — not universally understood

### Document structure
- Start by defining scope: what the document covers and what it does NOT cover
- State prerequisites (what the reader must know/do before reading)
- Put key information first — a busy reader may only read the first section
- Connect new concepts to familiar ones: "Similar to X, but with Y"

### Punctuation
- Oxford (serial) comma: "C++, Python, Java, and JavaScript" — always use
- Semicolon: joins two closely related, complete sentences
- Em dash (—): stronger pause than a comma; used to set off digressions
- Colon: before a list or table
- Parentheses: for secondary information; minimize their use

## Patterns to apply (before → after)

| Before | After |
|---|---|
| "Protocol Buffers...later, protobufs..." | "**Protocol Buffers** (**protobufs**) are..." |
| "This is a variable." | "This `counter` variable stores..." |
| "An error is generated." | "The system generates an error." |
| "There is a prop called `size`." | "The `size` prop controls..." |
| "The component runs screamingly fast." | "The component renders in under 16ms." |
| "at this point in time" | "now" |
| "is able to" | "can" |

## Patterns to avoid

- Synonymizing terms ("Button", "control", "element" for the same component)
- Undefined acronyms on first use
- Pronouns without a clear referent ("It renders the..." — what renders?)
- Passive voice when the actor is known
- "There is / There are" at the start of a sentence
- Subjective adjectives: "easy", "simple", "powerful", "fast" without data

## Application in DS

- **Component names**: pick one name and use it consistently (e.g., "Button", not "Button/button/Btn")
- **Props**: define on first introduction; use inline code (`size`, `variant`)
- **Prop descriptions**: one prop = one sentence; start with a third-person verb ("Controls the...", "Specifies the...", "Enables...")
- **Variant lists**: bulleted list with parallel structure
- **Usage steps**: numbered list, each step starts with an imperative verb
- **Examples**: "Use `Button` when..." instead of "You can use `Button` when..."
- **Visual states (hover, active, disabled, focus)**: describe by behavior, not appearance — "The disabled state prevents user interaction" not "The disabled state looks grayed out"
- **Do / Don't patterns**: "Use `Button` for form submission. Don't use `Button` for navigation — use `Link` instead."
- **Component accessibility section**: "The `Button` component uses `role="button"` and responds to keyboard events." (active voice, specific attributes)
