# Tech Writing One — Reference

Source: https://developers.google.com/tech-writing/one

## Kluczowe zasady

### Słowa i terminologia
- Zdefiniuj nowe terminy przy pierwszym użyciu lub linkuj do istniejącej definicji
- Używaj tego samego terminu konsekwentnie w całym dokumencie — nie synonimizuj
- Akronimy: przy pierwszym użyciu napisz pełną formę + akronim w nawiasie, pogrubiając oba: **Protocol Buffers** (**PB**)
- Definiuj akronim tylko jeśli: (a) jest znacznie krótszy niż pełna forma AND (b) pojawia się wielokrotnie
- Zaimki (it, they, this, that) — umieszczaj max. 5 słów od rzeczownika, do którego się odnoszą; przy wątpliwości powtórz rzeczownik zamiast używać zaimka

### Aktywny vs. pasywny głos
- Schemat aktywnego: **podmiot + czasownik + obiekt** ("The compiler generates an error")
- Schemat pasywnego: obiekt + forma "be" + imiesłów ("An error is generated by the compiler")
- Aktywny głos: mniej słów, jasne kto wykonuje akcję, łatwiejszy do czytania
- Pasywny można stosować wyjątkowo gdy wykonawca akcji jest nieistotny lub nieznany
- Zdania rozkazujące ("Open the file", "Click Save") są aktywne — domyślny podmiot to "you"

### Mocne czasowniki
- Unikaj generycznych: is/are/was, occur, happen
- Zastąp mocnym, precyzyjnym czasownikiem:
  - "An error occurs" → "The system raises an error"
  - "There is a bug" → "The component contains a bug"
  - "We are careful" → "We carefully validate"
- Unikaj "There is / There are" — przesuń właściwy podmiot na początek zdania

### Krótkie, jasne zdania
- Jedno zdanie = jedna myśl (jak jedna instrukcja w kodzie)
- Zastąp rozwlekłe frazy zwięzłymi:
  - "at this point in time" → "now"
  - "is able to" → "can"
  - "provides a detailed description of" → "describes"
  - "causes the triggering of" → "triggers"
- Zdania podrzędne: trzymaj je tylko gdy rozwijają główną myśl; jeśli zmieniają temat → nowe zdanie
- **That** (bez przecinka) = klauzula niezbędna: "calculations that avoid linear algebra"
- **Which** (z przecinkiem) = klauzula dodatkowa: "Python, which Guido invented, ..."
- Unikaj przymiotników i przysłówków subiektywnych; zastąp danymi liczbowymi:
  - "runs extremely fast" → "runs 225% faster"
- *Uwaga: reguła that/which dotyczy angielskiego; przy pisaniu po polsku stosuj odpowiednik zdania względnego z "który/która" z odpowiednią przecinkową strukturą*

### Listy i tabele
- **Lista punktowana** = elementy bez określonej kolejności; zmiana kolejności nie zmienia sensu
- **Lista numerowana** = kroki lub sekwencje; zmiana kolejności zmienia sens
- **Równoległość** — wszystkie elementy listy muszą mieć tę samą strukturę gramatyczną (np. wszystkie zaczynają się od czasownika)
- Listy numerowane: zacznij każdy element od trybu rozkazującego ("Open", "Configure", "Run")
- Każda lista i tabela wymaga zdania wprowadzającego zakończonego dwukropkiem:
  - "The component accepts the following props:"
- Tabele: krótkie komórki (max. 2 zdania), spójne typy danych w kolumnach, nagłówki kolumn

### Akapity
- Pierwsze zdanie akapitu = najważniejsze; musi wskazywać temat akapitu
- Jeden akapit = jeden temat; usuń zdania odbiegające od tematu
- Długość: 3–5 zdań (optymalnie); unikaj "ścian tekstu" (>7 zdań)
- Framework akapitu: **Co** (informacja) → **Dlaczego** (znaczenie) → **Jak** (zastosowanie)

### Odbiorcy
- Formuła: `dokumentacja = (wiedza potrzebna odbiorcy) − (wiedza, którą już ma)`
- Zidentyfikuj role, doświadczenie, znajomość projektu
- Unikaj "klątwy wiedzy" — nie zakładaj, że odbiorca wie co wiesz Ty
- Używaj prostego języka (wielu czytelników używa tłumacza); unikaj idiomów i kulturowych metafor
- Unikaj sportowych lub kulturowych odniesień (baseball, cricket) — nie są zrozumiałe globalnie

### Struktura dokumentu
- Zacznij od zdefiniowania zakresu: co dokument obejmuje i czego NIE obejmuje
- Podaj wymagania wstępne (co czytelnik musi wiedzieć/zrobić przed czytaniem)
- Kluczowe informacje umieszczaj na początku — zajęty czytelnik może przeczytać tylko pierwszą sekcję
- Łącz nowe koncepcje ze znajomymi: "Podobnie jak X, ale z Y"

### Interpunkcja
- Przecinek Oxford (serialowy): "C++, Python, Java, and JavaScript" — używaj zawsze
- Średnik: łączy dwa silnie powiązane, kompletne zdania
- Myślnik (—): mocniejsza pauza niż przecinek; do wyodrębnienia dygresji
- Dwukropek: przed listą lub tabelą
- Nawiasy: dla drugorzędnych informacji; minimalizuj ich użycie

## Wzorce do stosowania (before → after)

| Przed | Po |
|---|---|
| "Protocol Buffers...later, protobufs..." | "**Protocol Buffers** (**protobufs**) are..." |
| "This is a variable." | "This `counter` variable stores..." |
| "An error is generated." | "The system generates an error." |
| "There is a prop called `size`." | "The `size` prop controls..." |
| "The component runs screamingly fast." | "The component renders in under 16ms." |
| "at this point in time" | "now" |
| "is able to" | "can" |

## Wzorce do unikania

- Synonimizowanie terminów ("Button", "control", "element" dla tego samego komponentu)
- Niezdefiniowane akronimy przy pierwszym użyciu
- Zaimki bez jasnego odniesienia ("It renders the..." — co renderuje?)
- Pasywny głos gdy wiadomo kto jest aktorem
- "There is / There are" na początku zdania
- Subiektywne przymiotniki: "easy", "simple", "powerful", "fast" bez danych

## Zastosowanie w DS

- **Nazwy komponentów**: wybierz jedną nazwę i używaj jej konsekwentnie (np. "Button", nie "Button/button/Btn")
- **Props**: definiuj przy pierwszym wprowadzeniu; używaj kodu inline (`size`, `variant`)
- **Opisy props**: jeden prop = jedno zdanie; zacznij od czasownika w 3. osobie ("Controls the...", "Specifies the...", "Enables...")
- **Listy wariantów**: lista punktowana z równoległą strukturą
- **Kroki użycia**: lista numerowana, każdy krok zaczyna się od trybu rozkazującego
- **Przykłady**: "Użyj `Button` gdy..." zamiast "Można użyć `Button` gdy..."
- **Stany wizualne (hover, active, disabled, focus)**: opisuj stanem, nie wyglądem — "The disabled state prevents user interaction" nie "The disabled state looks grayed out"
- **Wzorce Do / Don't**: "Use `Button` for form submission. Don't use `Button` for navigation — use `Link` instead."
- **Sekcja dostępności komponentu**: "The `Button` component uses `role="button"` and responds to keyboard events." (aktywny głos, konkretne atrybuty)
